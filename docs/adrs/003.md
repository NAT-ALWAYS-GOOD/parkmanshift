# ADR 003: Choix de la Stack Technologique (Backend & Frontend)

**Date:** 30 mars 2026

**Statut:** ACCEPTÉ

**Contributeur(s):** Noam DE MASURE, Tom BOURLARD, Anaelle EA

## Contexte

Suite à la décision de concevoir une application Web (ADR 001), il nous faut désigner les technologies et frameworks qui soutiendront le "Parking Reservation system".

La solution doit être déployable en conteneurs Docker, facilement testable, et l'architecture complète doit être transmissible aux futures équipes informatiques de l'entreprise. De plus, l'équipe de développement actuelle possède une forte expertise sur Java (Spring Boot) et sur Vue.js (TypeScript).

## Solutions envisagées

### Frontend (Client Web) :
1. React (avec TypeScript) : L'écosystème le plus utilisé sur le marché. Très flexible, mais impose souvent de multiplier les dépendances externes pour le routing et la de gestion d'état centralisé.
2. Angular : Excellent framework d'entreprise, strictement typé, imposant une structure rigoureuse. La courbe d'apprentissage est lente et le poids du développement initial peut ralentir la sortie d'un premier prototype en quelques heures.
3. Vue.js (avec TypeScript) : Courbe d'apprentissage en douceur, extrêmement performant et doté d'une syntaxe déclarative très claire. Il excelle pour itérer et livrer très rapidement des interfaces complexes tout en gardant un code maintenable et un typage fort.

### Backend (Serveur & API REST) :
1. Node.js (NestJS / Express) : Utiliser du TypeScript de bout en bout aurait permis d'avoir un langage unique pour toute l'équipe et potentiellement de partager du code de modèles entre front et back.
2. C# (.NET Core) : Très puissant et adapté aux milieux d'entreprise avec des fonctionnalités solides d'injection de dépendances et une grande fiabilité, mais demanderait un temps d'adaptation important pour l'équipe en dehors de toute expertise existante.
3. Java avec Spring Boot : Le framework standard par excellence en entreprise pour la création d'APIs et de micro-services évolutifs. Apporte une multitude d'outils (Spring Data, Spring Security) et s'intègre facilement dans un écosystème complexe en conteneurs ou asynchrone (messagerie liée à l'ADR 002).

## Décision

Nous avons opté pour le modèle d'architecture applicative suivant :
- API Backend : Java avec Spring Boot 
- Interface Frontend : Vue.js avec TypeScript 

Ces choix se sont naturellement imposés pour maximiser l'efficacité de l'équipe (en tirant parti de ses compétences acquises) tout en répondant aux exigences de l'entreprise.

## Conséquences

### Bénéfices

- Time-to-Market et Vitesse : L'expertise de l'équipe sur Spring Boot et Vue garantit le respect des livraisons intensives imposées par les sprints de l'outil.
- Propreté et maintenabilité du code : Spring Boot force une architecture en couches claire (Controller, Service, Repository), simplifiant la gestion des logs, de l'historique et des règles métier. Vue associé à TypeScript apporte fiabilité des données lors de l'intégration continue.
- Passation aux futures équipes fluides : Ces technologies font l'objet d'un très fort consensus au sein des standards actuels du marché pour le support aux entreprises, facilitant l'onboarding pour des développeurs ultérieurs.

### Inconvénients

- Isolement des langages : L'absence d'environnement technique natif commun exigera d'écrire et de maintenir les modèles de données (DTOs et contrats de requêtes/réponses) en deux fois (une fois en objet Java, une fois en interface TypeScript côté front). 
- Poids applicatif et démarrage du conteneur : Un conteneur exécutant une application Java pèse potentiellement plus lourd et démarre de manière légèrement plus lente au runtime qu'un équivalent en Node.js, ce qui se manifestera lors du déploiement Docker.

## Suivi

- Initialiser les deux applications avec les technologies choisies.
- Prendre une décision sur l'utilisation de l'API First.
