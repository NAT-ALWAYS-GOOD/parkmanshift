# ADR 002: Architecture Événementielle et Gestion des Notifications

**Date:** 30 mars 2026

**Statut:** ACCEPTÉ

**Contributeur(s):** Noam DE MASURE, Tom BOURLARD, Anaelle EA

## Contexte

Dans le cadre de l'application de réservation de places de parking, les règles métiers précisent que :
- "Lorsqu'un utilisateur (employé ou manager) effectue une réservation, un message doit être envoyé dans une file d'attente (queue) afin d'être traité par une autre application qui enverra un e-mail de confirmation."

L'envoi des e-mails, pris en charge par une application tierce, peut parfois connaitre des ralentissements ou des interruptions de service temporaires. Il est crucial que l'expérience utilisateur et les performances globales du système de réservation ne soient pas impactées par le service externe de notification.

## Solutions envisagées

1. Appel synchrone (REST/gRPC)
   - L'application de réservation appelle directement l'API du service de notification de manière synchrone, en attendant la confirmation de l'envoi de l'e-mail avant de répondre à l'utilisateur.

2. Architecture événementielle asynchrone avec Message Broker / File d'attente (Message Queue)
   - L'application principale publie un événement (ex: `ReservationCreated`) dans un système de messagerie (ex: RabbitMQ, AWS SQS, Kafka, Redis Storage). L'application tierce chargée des notifications agit comme un "consommateur", dépilant ces événements à son propre rythme de manière asynchrone.

3. Base de données partagée (Polling)
   - L'application enregistre les e-mails à envoyer dans une table spécifique de la base de données (Outbox Pattern basique). L'application tierce interroge régulièrement (polling) cette table pour trouver puis traiter les nouveaux messages.

## Décision

Nous avons opté pour l'Architecture événementielle asynchrone avec file d'attente (Solution 2). 

Cette approche permet de découpler totalement le système principal de réservation du système secondaire d'envoi d'e-mails. Si le service d'e-mailing est indisponible ou lent, les événements de réservation s'accumuleront simplement dans la file et seront traités ultérieurement, sans jamais bloquer l'utilisateur ou empêcher sa réservation en base. Elle répond aussi directement à la contrainte explicite formulée dans les besoins du produit ("a message to a queue should be sent").

## Conséquences

### Bénéfices

- Résilience accrue : Le système de réservation reste fonctionnel même si le service d'e-mail de confirmation est en panne. Les messages de notifications ne sont pas perdus (mécanisme de rétention de la file).
- Temps de réponse rapide : Publier un message dans une file d'attente est quasi instantané, offrant une meilleure expérience utilisateur.
- Scalabilité et Découplage : L'application de notification peut évoluer ou être mise à l'échelle indépendamment.
- Extensibilité : Très facile d'ajouter de nouveaux systèmes qui réagiront à l'événement de réservation (par exemple, pour mettre à jour automatiquement le calendrier Outlook de l'employé) en ajoutant de nouveaux consommateurs sans impacter l'application principale.

### Inconvénients

- Complexité de l'infrastructure : Ajout d'une nouvelle brique technologique (broker de messages conteneurisé) à configurer et maintenir.
- Traçabilité asynchrone : Le suivi des bugs et traçage des requêtes "de bout en bout" devient un peu plus complexe et nécessitera un trace ID partagé (Correlation ID).
- Éventualité de messages dupliqués : Le consommateur des notifications devra se prémunir contre les envois en doublon (idempotence de la réception, concept "At-least-once delivery").

## Suivi

- Choisir la technologie exacte du Message Broker (ex: RabbitMQ en conteneur Docker).
- Définir le format et le schéma précis des messages échangés (données de l'événement).
- Mettre en place le producteur d'événement dans l'application de réservation.
