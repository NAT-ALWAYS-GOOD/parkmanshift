# ADR 004: Approche API-First et Génération de Code

**Date:** 30 mars 2026

**Statut:** ACCEPTÉ

**Contributeur(s):** Noam DE MASURE, Tom BOURLARD, Anaelle EA

## Contexte

Dans l'ADR 003, nous avons décidé d'utiliser des langages différents pour le backend (Java avec Spring Boot) et le frontend (Vue.js avec TypeScript). L'un des inconvénients majeurs soulevés de ce choix était le risque de divergence et la duplication de code concernant les contrats d'interfaces et les modèles de données (DTOs, objets de requêtes et de réponses).

## Solutions envisagées

1. Code-First (Approche traditionnelle implicite)
   - Le backend écrit les contrôleurs et les entités en Java, puis un outil génère la documentation OpenAPI à la volée. Le frontend doit ensuite consulter manuellement cette documentation pour recréer (et maintenir à jour) de son côté les interfaces TypeScript associées. 
   - Inconvénient majeur : Risque d'erreur humaine et duplication des structures de données.

2. API-First (Design-first) avec Spécification OpenAPI
   - L'équipe rédige d'abord le contrat d'API (un fichier OpenAPI en YAML ou JSON) qui recense toutes les routes, les schémas et tous les types de sécurité avant même d'écrire du code métier.
   - Ce document devient la source de vérité absolue. Les code source des interfaces (TypeScript) et parfois même des modèles serveurs (Java) sont purement générés automatiquement via des outils au moment du build.

## Décision

Nous avons opté pour le modèle de conception API-First.

Toute nouvelle fonctionnalité de notre Parking Reservation system fera systématiquement l'objet d'une conception préalable dans un contrat OpenAPI. Ce contrat pilotera les développements.

## Conséquences

### Bénéfices

- Documentation incluse et à jour : La documentation de l'API est maintenue automatiquement et devient un artefact livrable naturel (très utile pour de futures équipes "Business Support").
- Zéro duplication de code : Élimination totale de l'inconvénient identifié à l'ADR 003 de devoir recoder les objets à la fois en Java et en TS. Les schémas sont écrits à un seul endroit.
- Sécurité et cohérence (Single Source of Truth) : Le contrat API devient la seule source de vérité.
- Parallélisation optimale : Le développeur Vue.js peut commencer à coder ses vues grâce à des bibliothèques de Mock basées sur le fichier de spécification OpenAPI en attendant que le développeur Spring Boot ne termine l'implémentation.

### Inconvénients

- Rédiger du YAML de spécification OpenAPI peut être rébarbatif avant de plonger dans le code et demande un temps d'adaptation technique.

## Suivi

- Rédiger le contrat d'API initial dans le dossier des spécifications architecturales lors de la conception.
- Mettre en place la chaine de génération de code de l'API côté Spring Boot.
- Mettre en place la chaine de génération des clients REST et des types TypeScript côté Vue.
